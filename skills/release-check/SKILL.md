# 릴리즈 체크

데모 전 또는 배포 전 최종 점검에서 사용한다.

## 필수 점검

1. 규칙 무결성.
- 하루 1회 인증 제한이 지켜진다.
- 주간 목표 3회(월~일)가 지켜진다.
- 3회 미만 시 벌금 10,000원이 적용된다.

2. API 정상 동작.
- `GET /api/health`는 `ok`를 반환한다.
- `POST /api/checkins`는 단건 등록이 가능하다.
- 동일 날짜 중복 인증은 거부된다.
- `GET /api/weekly-status`는 pass/fail과 벌금을 정확히 반환한다.

3. UI 정상 동작.
- 첫 화면에서 백엔드 상태가 보인다.
- 규칙 요약이 보인다.
- API 실패 시 에러 메시지가 보인다.

4. 실행 문서 무결성.
- 루트 `README.md` 실행 명령이 현재 코드와 일치한다.

5. Git 무결성.
- 릴리즈 체크 시작 시 브랜치가 clean 상태다.
- 역할 브랜치 머지 순서(planner, backend, frontend, qa)가 지켜진다.
- 완료된 역할 task 커밋이 origin에 push되어 있다.
- `main`에는 리뷰된 변경만 포함된다.

## 결과 보고 형식

- 항목별로 Passed/Failed와 실패 지점을 정확히 기록한다.
- 메모는 짧고 실행 가능하게 작성한다.

## 가드레일

- 요청 없이는 릴리즈 체크 중 리팩터링하지 않는다.
- 문제 해결은 가능한 가장 작은 수정으로 처리한다.
- 로컬에만 존재하고 push되지 않은 작업이 있으면 릴리즈를 중단한다.
- 변경 파일이 문서(`*.md`)만이면 테스트 실행을 필수로 요구하지 않는다.
